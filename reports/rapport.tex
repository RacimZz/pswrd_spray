% ── 1. Introduction ───────────────────────────────────────────────────────────
\section{Introduction}

Le \textit{password spraying} est une attaque d'authentification consistant à
tester un petit nombre de mots de passe courants sur un grand nombre de comptes
distincts. Contrairement au brute force classique, cette approche vise à rester
sous les seuils de détection habituels (verrouillage de compte après $N$ échecs),
rendant son identification particulièrement difficile avec des méthodes classiques.

Ce projet propose un système de détection basé sur l'analyse comportementale des
flux d'authentification et les méthodes de détection d'anomalies non supervisées.
L'objectif est d'identifier des patterns suspects au niveau des adresses IP sources
sans nécessiter de données labellisées à l'entraînement.

\subsection*{Contributions}
\begin{itemize}
    \item Implémentation d'un pipeline de détection d'anomalies avec Isolation Forest
    \item Évaluation comparative sur deux datasets publics (CIC-IDS-2017, UNSW-NB15)
    \item Analyse des limites de la détection flux-par-flux face aux attaques discrètes
    \item Démonstration de l'apport de l'agrégation comportementale par IP
\end{itemize}

% ── 2. Données ────────────────────────────────────────────────────────────────
\section{Données}

\subsection{CIC-IDS-2017}

Le dataset CIC-IDS-2017 \cite{cicids2017} est produit par le Canadian Institute
for Cybersecurity. Il contient 2\,830\,743 flux réseau capturés sur une semaine,
représentant du trafic normal ainsi que 14 types d'attaques différentes.
Chaque flux est décrit par 78 features numériques extraites au niveau réseau
(durée, volume de paquets, flags TCP, etc.) et une colonne \texttt{Label}.

Ce dataset est particulièrement adapté à l'évaluation rigoureuse car le split

% ── 3. Méthode ────────────────────────────────────────────────────────────────
\section{Méthode}

\subsection{Isolation Forest}

Isolation Forest \cite{iforest} est un algorithme de détection d'anomalies
non supervisé. Il construit un ensemble d'arbres de décision aléatoires en
isolant les observations : les anomalies, peu nombreuses et atypiques, sont
isolées en moins de découpages que les points normaux.

Le score d'anomalie d'un point $x$ est défini par :
\[
s(x, n) = 2^{-\frac{\mathbb{E}[h(x)]}{c(n)}}
\]
où $h(x)$ est la longueur de chemin moyenne dans les arbres et $c(n)$ est
la longueur moyenne attendue pour $n$ observations.

\subsection{Pipeline}

Le pipeline implémenté suit les étapes suivantes :

\begin{enumerate}
    \item \textbf{Chargement} des données brutes (CSV ou Parquet)
    \item \textbf{Nettoyage} : remplacement des valeurs infinies et NaN par 0
    \item \textbf{Sélection} des features numériques uniquement
    \item \textbf{Normalisation} par \texttt{StandardScaler} (moyenne 0, écart-type 1)
    \item \textbf{Entraînement} sur le trafic normal uniquement (non supervisé)
    \item \textbf{Scoring} : calcul du score d'anomalie sur l'ensemble du test set
    \item \textbf{Décision} : seuil basé sur le paramètre \texttt{contamination}
\end{enumerate}

\subsection{Agrégation comportementale par IP}

Pour améliorer la détection des attaques discrètes, nous proposons une
agrégation des flux par adresse IP source sur des fenêtres temporelles de 10
minutes. Les features calculées sont :

\begin{itemize}
    \item $n\_flows$ : nombre total de flux dans la fenêtre
    \item $n\_dst\_ports$ : nombre de ports de destination distincts
    \item $fail\_rate$ : taux d'échec des connexions
    \item $flows\_per\_min$ : débit de connexions par minute
    \item $port\_entropy$ : entropie des ports de destination
\end{itemize}

Cette approche s'inspire des méthodes Risk-Based Authentication (RBA)
\cite{rba} qui agrègent le comportement par utilisateur pour détecter
des connexions anormales.

% ── 4. Résultats ──────────────────────────────────────────────────────────────
\section{Résultats}

\subsection{Optimisation des hyperparamètres}

Une recherche sur grille a été conduite sur le dataset UNSW-NB15 afin
d'identifier la configuration la plus performante pour chaque modèle. Les
hyperparamètres testés pour Isolation Forest étaient
\texttt{n\_estimators} $\in \{100, 200, 300\}$,
\texttt{max\_samples} $\in \{0.5, 0.8, 1.0\}$ et
\texttt{contamination} $\in \{0.3, 0.4, 0.499\}$.

Les performances ont été évaluées à l'aide de trois métriques
complémentaires : la \textbf{precision}, le \textbf{recall} et le
\textbf{F1-score}. La precision mesure la proportion d'alertes émises par
le modèle qui correspondent réellement à des attaques ; ici, une precision
de 100\% signifie que toutes les observations signalées comme anormales
dans cette évaluation étaient effectivement malveillantes. Le recall mesure
la proportion des attaques réellement présentes dans le jeu de test qui ont
été détectées. Enfin, le F1-score est la moyenne harmonique entre precision
et recall : il résume donc le compromis global entre capacité de détection
et fiabilité des alertes.

\begin{table}[H]
\centering
\caption{Comparaison des modèles — meilleure configuration}
\begin{tabular}{lccc}
\toprule
\textbf{Modèle} & \textbf{F1} & \textbf{Recall} & \textbf{Precision} \\
\midrule
Isolation Forest & \textbf{0.743} & \textbf{82.8\%} & 100\% \\
One-Class SVM    & 0.736          & 85.6\%          & 100\% \\
Local Outlier Factor & 0.540      & 51.6\%          & 100\% \\
\bottomrule
\end{tabular}
\end{table}

Les résultats montrent qu'Isolation Forest obtient le meilleur
\textbf{F1-score} avec une valeur de 0.743, ce qui traduit le meilleur
équilibre global entre détection et précision parmi les modèles comparés.
Même si One-Class SVM atteint un recall légèrement supérieur
(85.6\% contre 82.8\%), son F1-score est un peu plus faible, ce qui
justifie le choix d'Isolation Forest comme modèle final. Local Outlier
Factor apparaît nettement moins performant, avec un recall limité à
51.6\%, ce qui signifie qu'il laisse échapper près d'une attaque sur deux
dans cette configuration.

Le meilleur modèle retenu est donc \textbf{Isolation Forest} avec
\texttt{n\_estimators=100}, \texttt{max\_samples=1.0} et
\texttt{contamination=0.499}. Le paramètre \texttt{n\_estimators}
correspond au nombre d'arbres aléatoires de la forêt, \texttt{max\_samples}
à la fraction des données utilisée pour construire chaque arbre, et
\texttt{contamination} à la proportion estimée d'anomalies dans les
données, utilisée pour calibrer le seuil de décision. Dans cette
configuration, le modèle atteint un F1-score de 0.743 et un recall de
82.8\% sur le jeu de test UNSW-NB15.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.75\textwidth]{figures/fig2_comparaison_modeles.pdf}
    \caption{Comparaison F1 et Recall des trois modèles testés}
\end{figure}

La figure compare directement les performances globales des trois modèles.
Elle met en évidence que One-Class SVM et Isolation Forest ont des niveaux
de recall proches, mais qu'Isolation Forest conserve le meilleur compromis
global grâce à son F1-score légèrement supérieur. Elle permet donc de
visualiser pourquoi le choix final ne repose pas uniquement sur le recall,
mais sur une évaluation plus équilibrée.

\subsection{Distribution des scores}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.75\textwidth]{figures/fig3_distribution_scores.pdf}
    \caption{Distribution des scores d'anomalie selon la classe réelle}
\end{figure}

Cette figure représente la distribution des scores d'anomalie produits par
le modèle pour les observations normales et pour les attaques. L'objectif
est de vérifier si le modèle attribue, en moyenne, des scores plus élevés
aux comportements malveillants qu'au trafic légitime.

On observe une séparation globalement satisfaisante entre les deux classes,
ce qui indique que le modèle parvient à distinguer une large partie des
attaques du trafic normal. Toutefois, un chevauchement partiel subsiste
entre les distributions : ce chevauchement correspond aux cas ambigus, où
certaines attaques ressemblent au trafic normal ou inversement. Ce sont ces
zones de recouvrement qui expliquent principalement les faux négatifs
restants, c'est-à-dire les attaques non détectées.

\subsection{Matrice de confusion}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{figures/fig4_confusion_matrix.pdf}
    \caption{Matrice de confusion du modèle final sur UNSW-NB15}
\end{figure}

La matrice de confusion fournit une lecture plus détaillée des résultats en
distinguant quatre cas : les vrais positifs, les vrais négatifs, les faux
positifs et les faux négatifs. Elle permet donc de comprendre non seulement
combien d'exemples sont bien classés, mais aussi la nature des erreurs du
modèle.

Dans notre cas, la precision de 100\% suggère l'absence, ou la quasi-absence,
de faux positifs sur cette évaluation : les alertes produites sont donc très
fiables. En revanche, le recall inférieur à 100\% montre qu'une partie des
attaques reste non détectée ; ces erreurs apparaissent dans la matrice sous
la forme de faux négatifs. La matrice confirme ainsi que la limite
principale du modèle n'est pas la génération de fausses alertes, mais la
détection incomplète de certains comportements malveillants.

\subsection{Courbe Precision-Recall}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.75\textwidth]{figures/fig5_precision_recall.pdf}
    \caption{Courbe Precision-Recall — modèle final}
\end{figure}

La courbe Precision-Recall évalue le comportement du modèle lorsque l'on
fait varier le seuil de décision appliqué au score d'anomalie. Elle montre
comment évolue la precision lorsque l'on cherche à augmenter le recall, et
inversement. Cette courbe est particulièrement pertinente dans le cas de la
détection d'intrusions, où les classes sont souvent déséquilibrées.

Une courbe située nettement au-dessus de la ligne de base traduit une bonne
capacité de séparation entre trafic normal et trafic malveillant. Dans notre
cas, cette figure permet d'analyser le compromis entre prudence et
sensibilité : un seuil plus strict réduit le nombre d'alertes mais risque
de manquer davantage d'attaques, tandis qu'un seuil plus permissif augmente
le rappel au prix d'un risque plus élevé de dégradation de la precision.
Elle complète donc utilement les métriques ponctuelles données dans le
tableau précédent.

\subsection{Recall par catégorie d'attaque}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.75\textwidth]{figures/fig6_recall_categories_unsw.pdf}
    \caption{Recall par catégorie d'attaque sur UNSW-NB15}
\end{figure}

Cette figure détaille le recall non plus globalement, mais séparément pour
chaque famille d'attaque présente dans UNSW-NB15. Elle est importante car
un bon recall moyen peut masquer de fortes disparités entre catégories :
certaines attaques peuvent être très bien détectées, tandis que d'autres
restent difficiles à identifier.

L'intérêt principal de cette visualisation est donc d'identifier les types
d'attaques les mieux capturés par l'approche comportementale et ceux pour
lesquels le modèle montre encore des limites. Une telle analyse est
essentielle dans un contexte de cybersécurité, car l'utilité pratique d'un
détecteur dépend non seulement de sa performance moyenne, mais aussi de sa
capacité à couvrir des scénarios d'attaque variés.

\subsection{Analyse flux brut vs agrégation comportementale}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.85\textwidth]{figures/fig1_recall_par_attaque.pdf}
    \caption{Recall par type d'attaque : CIC-IDS flux brut vs UNSW-NB15}
\end{figure}

Cette analyse compare deux approches de détection : une approche fondée sur
les caractéristiques de chaque flux pris isolément, et une approche
d'agrégation comportementale où les événements sont regroupés par IP sur une
fenêtre temporelle de 10 minutes. L'objectif est d'évaluer si l'observation
du comportement global d'une source permet de mieux détecter les attaques
lentes, répétitives ou distribuées dans le temps.

L'analyse comparative sur CIC-IDS-2017 révèle une limite fondamentale de
l'approche flux-par-flux face aux attaques discrètes. Le tableau ci-dessous
résume les résultats :

\begin{table}[H]
\centering
\caption{Recall selon l'approche de détection sur CIC-IDS-2017}
\begin{tabular}{lcc}
\toprule
\textbf{Type d'attaque} & \textbf{Flux brut} & \textbf{Agrégation IP/10min} \\
\midrule
DoS GoldenEye   & 96.2\% & --- \\
DoS Slowhttptest & 95.7\% & --- \\
SSH-Patator     & 49.9\% & 50.0\% \\
FTP-Patator     & 1.3\%  & \textbf{50.0\%} \\
PortScan        & 0.5\%  & \textbf{91.1\%} \\
\bottomrule
\end{tabular}
\end{table}

Les résultats montrent que les attaques très visibles au niveau d'un flux
individuel, comme les attaques DoS, sont déjà bien détectées sans
agrégation, avec des recalls supérieurs à 95\%. En revanche, pour des
attaques plus progressives ou réparties sur plusieurs connexions, comme
FTP-Patator et PortScan, l'analyse flux-par-flux devient insuffisante. Dans
ces cas, l'agrégation comportementale apporte un gain majeur, car elle
reconstruit une vue d'ensemble du comportement de l'attaquant à l'échelle
de l'IP et du temps.

L'effet est particulièrement marqué pour FTP-Patator, dont le recall passe
de 1.3\% à 50.0\%, soit un gain d'environ \textbf{38$\times$}. Pour
PortScan, l'amélioration est encore plus nette en valeur absolue, avec un
passage de 0.5\% à 91.1\%. Ces résultats confirment que les attaques lentes
et répétitives ne sont pas toujours détectables à partir d'un flux isolé,
mais deviennent beaucoup plus visibles lorsqu'on analyse le comportement
global d'une même IP sur une fenêtre temporelle.


% ── 5. Discussion ─────────────────────────────────────────────────────────────
\section{Discussion}

\subsection{Limites}

\textbf{Contamination fixe.} Le paramètre \texttt{contamination} contrôle
le seuil de décision directement. Sur CIC-IDS agrégé, fixer
\texttt{contamination} au taux réel d'attaques (31.6\%) produit un recall
artificiellement élevé. Les résultats UNSW-NB15, avec un split train/test
indépendant, sont plus fiables.

\textbf{Précision à 100\%.} La précision parfaite observée est un artefact
de la métrique sur ce type de dataset déséquilibré --- elle ne reflète pas
les performances en production où le taux de faux positifs est critique.

\textbf{Absence d'IP réelles dans CIC-IDS CSV.} Les fichiers CSV de
CIC-IDS ne contiennent pas les adresses IP sources, ce qui a nécessité
leur simulation pour l'agrégation comportementale.

\subsection{Perspectives}

\begin{itemize}
    \item Tester sur CIC-IDS-2018 qui contient des attaques brute force SSH/FTP plus récentes
    \item Intégrer une composante temporelle (LSTM ou sliding window) pour améliorer
          la détection des attaques lentes
    \item Comparer avec des méthodes supervisées (Random Forest, XGBoost) pour
          quantifier le coût du non-supervisé
\end{itemize}

% ── 6. Conclusion ─────────────────────────────────────────────────────────────
\section{Conclusion}

Ce projet démontre qu'Isolation Forest est une approche compétitive pour la
détection d'intrusions non supervisée, atteignant 82.8\% de recall et un F1
de 0.743 sur UNSW-NB15. L'analyse comparative sur CIC-IDS-2017 révèle que
l'agrégation comportementale par IP est indispensable pour détecter les
attaques discrètes de type password spraying : FTP-Patator passe de 1.3\%
à 50\% de recall avec cette approche.

Ces résultats motivent une architecture hybride combinant détection
flux-par-flux pour les attaques volumiques et agrégation comportementale
pour les attaques lentes, à l'image des systèmes RBA déployés en production.
